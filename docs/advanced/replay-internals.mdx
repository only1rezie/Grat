The replay engine is the core component that powers Tier 2 and Tier 3 diagnostics. It allows Prism to show you exactly what happened inside a contract invocation on mainnet.

## Two-Path State Reconstruction

Prism uses two different strategies to reconstruct the ledger state, depending on how old the transaction is:

### Hot Path (Recent Transactions)
For transactions within the last ~50,000 ledgers, Prism uses the Soroban RPC `getLedgerEntries` method. This is fast and efficient for debugging active development.

### Cold Path (Older Transactions)
For older transactions, Prism fetches ledger snapshots from **Stellar History Archives**. It uses a captive core instance to replay ledgers up to the point of the transaction, ensuring 100% state accuracy.

## Instrumented Sandbox

Prism does not use a standard Soroban host. Instead, it uses a modified `soroban-env-host` with **trace hooks** injected at the host function boundary.

<Info>
  **Why host function boundaries?** Production WASM blobs have DWARF debug symbols stripped. Tracing at the host function level ensures compatibility with every contract on the network without requiring original source code.
</Info>

## Key Components

1. **State Reconstructor**: Orchestrates fetching ledger entries from RPC or Archives.
2. **Execution Sandbox**: The instrumented environment where the transaction is re-simulated.
3. **Trace Collector**: Aggregates raw events into a hierarchical tree (Contract → Host Call → Event).
4. **State Differ**: Compares the ledger state before and after execution to show changes.
